# CBTI 개선 및 발전 로드맵

작성일: 2026-05-07

## 1. 방향성 요약

현재 프로젝트는 사용자가 테스트를 완주하고 결과를 공유하는 MVP 핵심 흐름이 구현되어 있습니다. 다음 단계에서는 단순 기능 추가보다 서비스의 신뢰도, 공유성, 운영성, 콘텐츠 확장성을 높이는 방향으로 발전시키는 것이 좋습니다.

우선순위가 높은 개선 축은 다음과 같습니다.

- 결과 공유 경험 강화
- 점수 계산과 제출 데이터 검증 강화
- 결과 콘텐츠의 설득력과 신뢰도 향상
- 운영 환경과 배포 설정 정리
- 관리자 통계 기능 고도화
- 코드 구조와 테스트 기반 개선

## 2. 우선순위 높은 개선 항목

### 2.1 공유 경험 개선

현재 결과 공유는 URL 복사 중심으로 구현되어 있습니다. 모바일 사용자를 고려하면 네이티브 공유 흐름을 우선 지원하는 것이 좋습니다.

개선 제안:

- `navigator.share`를 사용해 모바일 네이티브 공유 우선 제공
- 네이티브 공유가 실패하거나 지원되지 않는 환경에서는 클립보드 복사로 fallback
- 카카오톡, 인스타그램 스토리 등에 맞는 공유 문구 또는 이미지 템플릿 제공
- 결과 페이지에 “나도 테스트 해보기”와 “내 결과 공유하기”의 동선을 더 명확히 배치
- 공유 링크 클릭 후 테스트 재진입률을 추적할 수 있도록 방문 경로 통계 강화

기대 효과:

- 모바일 공유 전환율 향상
- 테스트의 자연 확산 가능성 증가
- 공유 결과 페이지의 재방문 가치 상승

### 2.2 점수 및 제출 검증 강화

현재 `/api/submissions`는 질문 ID와 선택지 ID의 존재 여부를 검증하지만, 특정 선택지가 실제로 해당 질문에 속하는지까지는 엄격히 검증하지 않습니다.

개선 제안:

- 각 답변의 `questionId`와 `optionId`가 실제 부모-자식 관계인지 검증
- 같은 질문에 대한 중복 답변 차단
- 제출된 `scores`를 그대로 신뢰하지 않고 서버에서 `answers` 기반으로 재계산
- 클라이언트가 보낸 `primaryPersona`와 서버 재계산 결과가 일치하는지 확인
- 유효하지 않은 제출은 통계 저장에서 제외

기대 효과:

- 조작된 데이터가 통계에 섞일 가능성 감소
- 관리자 통계 신뢰도 향상
- 향후 캠페인 운영 시 데이터 품질 확보

### 2.3 결과 신뢰도 향상

현재 결과는 대표 유형과 보조 성향, 점수 분포를 보여줍니다. 여기에 점수 차이와 복합 성향을 더 자연스럽게 해석해 주면 결과 만족도가 높아질 수 있습니다.

개선 제안:

- 대표 성향, 보조 성향, 약한 성향을 구분해서 표시
- 대표 유형과 2위 유형의 점수 차이가 작을 때 “복합형 성향” 안내 제공
- 동점 처리 기준을 내부 로직으로만 두지 않고 결과 문구에 자연스럽게 반영
- 유형별 “잘 맞는 공동체 분위기” 추가
- 유형별 “주의하면 좋은 신앙 습관” 추가
- 유형별 “함께 잘 어울리는 유형” 또는 “다른 유형 이해하기” 콘텐츠 추가

기대 효과:

- 결과가 단정적으로 보이는 문제 완화
- 사용자의 자기 이해 경험 강화
- 공유하고 싶은 결과 콘텐츠 증가

### 2.4 운영 환경 정리

현재 `.env.example`에는 핵심 환경변수가 정리되어 있지만, `layout.tsx`에서 사용하는 `NEXT_PUBLIC_GA_ID`는 포함되어 있지 않습니다. 또한 `siteUrl`은 코드에 하드코딩되어 있습니다.

개선 제안:

- `.env.example`에 `NEXT_PUBLIC_GA_ID` 추가
- `layout.tsx`의 `siteUrl`을 `NEXT_PUBLIC_SITE_URL` 기반으로 정리
- 배포 체크리스트 문서 추가
- Supabase 설정 방법 문서화
- 관리자 비밀번호와 세션 시크릿 설정 방법 문서화
- 운영 환경과 로컬 개발 환경의 차이를 명확히 정리

기대 효과:

- 배포 실수 감소
- 운영자 온보딩 시간 단축
- 환경 변경에 강한 구조 확보

## 3. 사용자 경험 개선

### 3.1 테스트 진행 UX

개선 제안:

- 선택 후 바로 넘어가기 전에 짧은 시각적 피드백 제공
- 이전 질문으로 돌아갔을 때 기존 선택 상태 표시
- 답변 변경 후 진행 흐름이 자연스럽게 이어지도록 처리
- 분석 로딩 화면에 부드러운 전환 효과 추가
- 테스트 중 브라우저 새로고침 또는 이탈 시 답변 복구 여부 검토

주의할 점:

- 테스트형 서비스에서는 진행 속도가 중요하므로, 피드백 애니메이션이 너무 길어지지 않게 해야 합니다.
- 질문 직접 이동 기능은 편리할 수 있지만, 현재 MVP에서는 복잡도를 늘릴 수 있으므로 우선순위는 낮습니다.

### 3.2 결과 화면 강화

개선 제안:

- “내 유형 한 줄 요약” 추가
- “이런 신앙 공동체와 잘 맞아요” 섹션 추가
- “성장할 때 조심하면 좋은 지점” 섹션 추가
- “다른 유형과의 관계 팁” 섹션 추가
- 추천 교파 문구의 민감성을 낮추고 안내 문구를 더 정중하게 개선
- 결과 카드의 이미지 저장 또는 공유 이미지 생성 기능 검토

### 3.3 접근성 개선

개선 제안:

- 키보드만으로 테스트 시작, 선택, 이전 이동, 공유가 가능한지 확인
- 버튼과 입력창의 focus 스타일 강화
- 색 대비 점검
- 캐릭터 이미지와 OG 이미지의 alt 문구 점검
- `alert()` 대신 화면 내 토스트 메시지 사용
- 모바일 화면에서 긴 결과 제목과 버튼 텍스트가 잘리지 않는지 확인

## 4. 콘텐츠 발전 방향

### 4.1 콘텐츠 버전 관리

질문이나 결과 문구가 바뀌면 과거 통계와 현재 통계가 섞일 수 있습니다. 이를 방지하려면 콘텐츠 버전 관리가 필요합니다.

개선 제안:

- `src/data/test.ts`에 `contentVersion` 상수 추가
- 제출 저장 시 `contentVersion`도 함께 저장
- 관리자 통계에서 버전별 필터 제공
- 콘텐츠 변경 시 `CONTENT_REVIEW.md`와 실제 데이터의 동기화 절차 마련

기대 효과:

- 질문 변경 이후에도 통계 해석 가능
- A/B 테스트나 콘텐츠 개선 실험의 기반 마련

### 4.2 복합형 결과

현재는 대표 유형 1개를 중심으로 결과를 보여줍니다. 향후에는 상위 2개 유형 조합을 기반으로 복합형 결과를 제공할 수 있습니다.

예시:

- `Intellectual + Liturgical`: 깊이와 전통을 중시하는 신앙형
- `Progressive + Social`: 환대와 실천을 함께 중시하는 신앙형
- `Orthodox + Relational`: 복음의 본질과 공동체 사랑을 함께 붙드는 신앙형
- `Charismatic + Relational`: 뜨거운 체험과 친밀한 돌봄을 중시하는 신앙형

주의할 점:

- 모든 조합을 한 번에 만들면 콘텐츠 관리 비용이 커집니다.
- 초기에는 자주 나오는 상위 조합 몇 개만 선별해 제공하는 방식이 현실적입니다.

### 4.3 콘텐츠 검수 워크플로우

개선 제안:

- `CONTENT_REVIEW.md`를 기준 문서로 유지
- 실제 `src/data/test.ts`와 문서 내용의 불일치를 확인하는 스크립트 추가
- 교파 추천 문구에 대한 별도 검수 체크리스트 마련
- 민감한 종교적 표현, 배제적 표현, 특정 교파에 대한 과도한 단정 표현 점검

## 5. 관리자 및 통계 기능 개선

### 5.1 CSV 다운로드

개선 제안:

- 제출 데이터 CSV 다운로드
- 유형별 집계 CSV 다운로드
- 일자별 집계 CSV 다운로드
- 문항별 선택지 분포 CSV 다운로드

주의할 점:

- 개인정보는 저장하지 않는 방향을 유지합니다.
- 다운로드 데이터에는 원본 방문자 ID가 아닌 해시값만 포함해야 합니다.

### 5.2 기간 필터

개선 제안:

- 최근 7일
- 최근 30일
- 전체 기간
- 사용자 지정 날짜 범위

기대 효과:

- 캠페인 전후 성과 비교 가능
- 특정 공유 이벤트나 배포 이후의 변화 확인 가능

### 5.3 통계 지표 확장

개선 제안:

- 방문 수 대비 완료율
- 유형별 비율
- 공유 결과 페이지 유입 수
- 경로별 방문 통계
- 날짜별 방문 수와 완료 수 비교
- 중간 이탈률 추적

중간 이탈률을 추적하려면 질문 진행 이벤트를 별도로 저장해야 합니다. 이 기능은 데이터 저장량이 늘어나므로 운영 목적이 분명할 때 도입하는 것이 좋습니다.

### 5.4 관리자 인증 강화

개선 제안:

- 관리자 로그아웃 버튼 추가
- 세션 만료 안내 추가
- 실패한 로그인 시도에 대한 간단한 rate limit 검토
- 운영 규모가 커질 경우 Supabase Auth 또는 Vercel Deployment Protection 도입

## 6. 기술 구조 개선

### 6.1 컴포넌트 분리

현재 `src/app/page.tsx`는 테스트 진행, 상태 관리, 결과 표시, 공유, 제출 전송을 모두 담당합니다. 기능이 더 커지기 전에 컴포넌트를 분리하면 유지보수가 쉬워집니다.

분리 후보:

- `IntroScreen`
- `QuestionScreen`
- `LoadingScreen`
- `ResultScreen`
- `ProgressBar`
- `OptionButton`
- `SharePanel`
- `ScoreChart`
- `InsightCard`

기대 효과:

- UI 변경 시 영향 범위 감소
- 테스트 작성 용이
- 결과 화면 확장에 유리

### 6.2 테스트 추가

우선 테스트 대상:

- `calculateScores`
- `resolvePrimaryPersona`
- `getClosePersonas`
- `getSortedScores`
- `getResultUrl`
- 제출 API payload validation
- 서버 측 점수 재계산 로직

점수 로직은 서비스 신뢰도의 핵심이므로 가장 먼저 테스트를 붙이는 것이 좋습니다.

### 6.3 빌드와 린트 점검

개선 제안:

- 의존성 설치 후 `npm run build` 확인
- TypeScript 타입 검사 확인
- `next lint` 스크립트가 Next.js 15 환경에서 적절한지 점검
- 필요 시 ESLint 설정과 스크립트 업데이트
- GitHub Actions 또는 Vercel 빌드 체크 도입

## 8. 가장 먼저 추천하는 작업

가장 먼저 진행할 만한 작업은 다음 세 가지입니다.

1. 공유 경험 개선
2. 점수 및 제출 검증 강화
3. 운영 환경변수와 배포 설정 정리

이 세 항목은 작업량에 비해 서비스 완성도와 운영 안정성을 크게 높일 수 있습니다. 특히 제출 검증 강화와 서버 측 점수 재계산은 통계 신뢰도와 직접 연결되므로 빠르게 반영하는 것이 좋습니다.
